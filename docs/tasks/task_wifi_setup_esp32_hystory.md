# История изменений по задаче: Перевод страницы настройки WiFi на ESP32

## Дата начала: 26 марта 2026 г.

### Исходное состояние
- Код WiFi Setup Server существует только для ESP8266
- Файлы: `lib\ETLTest\etl_wifi_setup.h`, `lib\ETLTest\etl_wifi_setup.cpp`, `lib\ETLTest\etl_wifi_setup_html.h`
- Использует библиотеки ESP8266: `ESP8266WiFi.h`, `ESP8266WebServer.h`, `ESP8266mDNS.h`

### Требуемые изменения
1. Добавить поддержку ESP32 через условную компиляцию
2. Использовать библиотеки ESP32: `WiFi.h`, `WebServer.h`, `ESPmDNS.h`
3. Учесть различия в API между ESP8266 и ESP32
4. Протестировать компиляцию для всех платформ

### План работ
1. Модифицировать заголовочный файл для поддержки обеих платформ
2. Модифицировать cpp файл для поддержки обеих платформ
3. Протестировать компиляцию для nodemcuv3 (ESP8266)
4. Протестировать компиляцию для esp32c3 (ESP32)
5. Протестировать компиляцию для esp32-wroom-32u (ESP32)

### Изменения

#### 1. etl_wifi_setup.h
- Добавлена условная компиляция для выбора библиотек WiFi и mDNS
- Добавлен алиас типа `etl_web_server_t` для совместимости `ESP8266WebServer` и `WebServer`
- Добавлена обёртка `#if defined(ESP8266) || defined(ESP32)` для всего кода
- Обновлены include директивы для ESP32

#### 2. etl_wifi_setup.cpp
- Изменено условие компиляции с `#if defined(ESP8266)` на `#if defined(ESP8266) || defined(ESP32)`
- Обновлён тип сервера с `ESP8266WebServer` на `etl_web_server_t`
- Добавлена условная компиляция для `MDNS.update()` (только ESP8266)
- Обновлена функция `get_mode()` для корректной работы на ESP32
- Обновлена функция `reboot()`: `ESP.reset()` для ESP8266, `ESP.restart()` для ESP32
- Обновлена функция `get_encryption_type()`: разные константы для ESP8266 и ESP32

#### 3. main.cpp
- Удалено ограничение `#ifndef ESP32` для использования WiFi сервера
- Исправлена ошибка с тернарным оператором и типами String

### Тестирование компиляции
Все конфигурации успешно скомпилированы:
- nodemcuv3 (ESP8266): SUCCESS
- esp32c3 (ESP32): SUCCESS
- esp32-wroom-32u (ESP32): SUCCESS

### Примечания
- На ESP32 `WiFi.softAPgetStationNum()` возвращает `uint8_t`, поэтому проверка изменена с `>= 0` на `> 0`
- На ESP32 `MDNS.update()` не существует, вызов удалён
- На ESP32 используются константы `WIFI_AUTH_*` вместо `ENC_TYPE_*`

## Исправления после тестирования (26 марта 2026 г.)

### Проблема 1: Disconnect перестает отвечать на запросы

**Симптомы:**
- При нажатии кнопки Disconnect сервер перестает отвечать
- Ошибка: `[WiFiClient.cpp:429] write(): fail on fd 49, errno: 113, "Software caused connection abort"`

**Причина:**
Ответ клиенту отправлялся **после** переключения WiFi в режим AP, что приводило к разрыву соединения до отправки ответа.

**Решение:**
В функции `handle_api_disconnect()` изменён порядок операций:
1. Сначала отправляется ответ клиенту
2. Затем небольшая задержка (100 мс)
3. После этого переключение WiFi в режим AP

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `handle_api_disconnect()`

### Проблема 2: Ошибка mDNS при перезапуске HTTP сервера

**Симптомы:**
```
[WiFiSetup] Restarting HTTP server after STA connection...
[112666][E][ESPmDNS.cpp:148] addService(): Failed adding service http.tcp.
```

**Причина:**
При перезапуске HTTP сервера после подключения к STA вызывался `MDNS.addService()`, но сервис уже был добавлен ранее. На ESP32 повторный вызов `addService()` для той же службы вызывает ошибку.

**Решение:**
Добавлен статический флаг `mdns_service_added` для отслеживания добавления сервиса. Сервис добавляется только один раз при первом запуске.

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `start_http_server()`

### Проблема 3: Настройки AP не сохраняются после перезапуска

**Симптомы:**
- При смене AP SSID и нажатии [Apply AP Settings] настройки применяются
- Точка доступа перезапускается с новым именем
- Но после перезагрузки устройства возвращаются старые значения

**Причина:**
В функции `handle_api_ap_settings()` настройки применялись через `setter'ы`, но не сохранялись в энергонезависимой памяти (LittleFS).

**Решение:**
Добавлен вызов `save_settings()` после применения настроек AP.

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `handle_api_ap_settings()`

### Проблема 4: Ошибка при повторном сохранении настроек

**Симптомы:**
- После [Apply AP Settings] настройки сохраняются успешно
- При нажатии [Save & Reboot] сохранение возвращает FAILED
- `[wifi::settings] save_config() result: FAILED`

**Причина:**
Библиотека FileData может возвращать `false` при повторном сохранении тех же данных (данные не изменились).

**Решение:**
В функции `handle_api_save()` изменена обработка неудачи сохранения - если сохранение не удалось, отправляется сообщение "Settings already saved" вместо ошибки.

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `handle_api_save()`

### Проблема 5: Поля AP настроек не обновляются из сохранённой конфигурации

**Симптомы:**
- Настройки загружаются из FS правильно (логи показывают `ap_ssid = ESP_Device_test`)
- Но на странице в полях ввода показываются значения по умолчанию (`ESP_Device_AP`)
- После изменения настроек и перезагрузки видно старое значение в UI

**Причина:**
JavaScript код загружал конфигурацию через `/api/config`, но не обновлял поля `apSsidInput` и `apPasswordInput` из загруженных данных.

**Решение:**
1. Добавлен `ap_password` в ответ `/api/config`
2. Обновлены функции `loadDeviceConfig()` и `applyDeviceConfig()` для сохранения и применения настроек AP

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `handle_api_config()`
- `lib/ETLTest/etl_wifi_setup_html.h` - функции `loadDeviceConfig()` и `applyDeviceConfig()`

### Проблема 6: Сервер не виден после перезагрузки с подключением к STA

**Симптомы:**
- После загрузки устройства с сохранёнными настройками STA подключение выполняется успешно
- Но сервер не виден по mDNS (http://espdevice.local)
- `get_mode()` возвращает `OFF`, IP адрес `0.0.0.0`

**Причина:**
1. После подключения к STA в `begin()` режим WiFi оставался в STA, а не переключался в AP+STA
2. Функция `get_mode()` использовала `WiFi.softAPgetStationNum()` для проверки активности AP, что на ESP32 возвращает количество подключённых станций, а не факт активности AP

**Решение:**
1. В `begin()` добавлено переключение в режим `WIFI_AP_STA` после успешного подключения к STA
2. `get_mode()` переписана для проверки через `WiFi.getMode()`

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функции `begin()` и `get_mode()`

### Проблема 7: После подключения к STA устройство переключается в режим AP

**Симптомы:**
- Подключение к sd_wifi успешно, получен IP 10.0.1.80
- Но затем устройство переключается в режим AP с IP 192.168.4.1
- Сервер не доступен из сети STA

**Причина:**
В функции `connect_to_sta()` после успешного подключения режим переключался на `previous_mode`, который был `WIFI_OFF` (режим по умолчанию после загрузки).

**Решение:**
В `connect_to_sta()` после подключения:
1. Переключение в режим `WIFI_AP_STA`
2. Запуск точки доступа, если она не была активна

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `connect_to_sta()`

### Проблема 8: Disconnect - сервер не отвечает после нажатия

**Симптомы:**
- При нажатии Disconnect появляется ошибка "Software caused connection abort"
- Сервер перестает отвечать на запросы

**Причина:**
На ESP32 переключение `WiFi.mode()` вызывает мгновенный разрыв соединения. Даже при отправке ответа перед переключением, ответ не успевает дойти до клиента.

**Решение:**
В `handle_api_disconnect()`:
1. Ответ отправляется клиенту
2. Увеличена задержка до 500 мс + `yield()` для обработки сетевых событий
3. Затем отключение от сети и переключение в режим AP

**Изменённые файлы:**
- `lib/ETLTest/etl_wifi_setup.cpp` - функция `handle_api_disconnect()`
